# 车载毫米波雷达 BSW 团队人员规划与外包策略

> **适用场景**：8×8 前向毫米波雷达，量产级开发
>  **范围说明**：仅覆盖 **BSW + 平台软件 + 系统集成 + 测试 + 功能/信息安全** 部分，雷达信号处理算法团队单独规划，不在本文范围内
>  **当前团队**：5 名嵌入式工程师
>  **目标**：从当前规模演进为可支撑量产的 BSW 团队

------

## 一、产品规格与团队压力评估

### 1.1 8×8 前雷达的技术特征

8 发 8 收（MIMO 等效 64 通道）属于当前主流 4D 成像雷达 / 高性能前雷达配置，对软件团队的要求显著高于普通角雷达：

| 维度         | 特征                                                         |
| ------------ | ------------------------------------------------------------ |
| 主流芯片方案 | TI AWR2243 级联 / AWR2944、NXP S32R45 / S32R294、Arbe、Uhnder 等 |
| 客户定位     | 主机厂前向感知（AEB / ACC / NOA）                            |
| 功能安全等级 | **至少 ASIL B(D) 分解，部分主机厂要求 ASIL D**               |
| 通信接口     | CAN-FD + 车载以太网（100/1000BASE-T1）+ SOME/IP              |
| 量产挑战     | 主机厂审核严苛、产线 EOL 标定复杂、客户问题单 SLA 短         |

### 1.2 量产期对团队的核心压力

1. **多版本回归压力**：每次软件版本更新需跑 HIL + 实车 + 暗室回归
2. **客户问题单 SLA**：量产后通常 24–72 小时响应窗口
3. **多变体并行**：一个平台可能衍生多个客户项目和 SOP 阶段
4. **产线问题插入**：随时打断正常迭代节奏
5. **功能安全审核**：主机厂派专家现场答辩，无法外包

## 三、分阶段招聘计划（5 → 8–9 人）

按车载前雷达典型项目周期（立项到 SOP 通常 18–24 个月）。

### 阶段 A：A 样阶段（项目启动 → 6 个月）

**目标**：跑通基础功能，CAN 通信能上车，能给主机厂演示

**5 人 → 7 人**，新招 2 人：

| 优先级 | 岗位                                | 主要职责                                |
| ------ | ----------------------------------- | --------------------------------------- |
| 1      | 测试 / HIL 工程师 ×1                | 搭建 HIL 环境、CANoe 工程、基础回归用例 |
| 2      | AUTOSAR 集成 + Bootloader 接口人 ×1 | 对接外包供应商、管住 BSW 交付质量       |

**关键考量**：

- 外包要在这个阶段启动，所以接口人必须先到位，否则交付物没人接
- 测试工程师早进来能从一开始建好测试基线，避免后期补测试债

### 阶段 B：B 样阶段（6 → 12 个月）

**目标**：功能完整，SOME/IP + 诊断 + Bootloader 全部 ready，启动功能安全活动

**7 人 → 8 人**，新招 1 人 + 启动外包咨询：

| 优先级 | 动作                             | 说明                                                         |
| ------ | -------------------------------- | ------------------------------------------------------------ |
| 1      | 测试 / HIL 工程师 ×1             | 补充人力，开始做自动化 CI，覆盖诊断 / Bootloader / SecOC 测试 |
| 2      | **启动功能安全咨询合同**（外包） | 由现有团队某人（建议平台负责人或架构师）兼任 FuSa owner      |

**关键考量**：

- B 样是功能安全活动高峰期，FuSa owner 必须明确
- 测试压力陡增（多版本回归 + 实车测试 + 暗室联调），1 人扛不住

### 阶段 C：SOP 前 6 个月（12 → 18 个月）

**目标**：PPAP、产线导入、客户审核、问题清零

**8 人 → 8–9 人**，按需新招：

| 优先级 | 岗位                       | 触发条件                     |
| ------ | -------------------------- | ---------------------------- |
| 可选   | 应用层 / 系统软件工程师 ×1 | 同时有 ≥2 个客户项目并行才补 |

**关键考量**：

- 这个阶段重点不是招人，是**让外包驻场**
- BSW 集成调试、Bootloader 刷写问题、UDS corner case 会集中爆发，外包工程师必须现场支持

### 阶段 D：SOP + 量产维护（18 个月之后）

**目标**：扛住客户问题单、产线问题、OTA 升级、新变体衍生

**保持 8–9 人，按业务扩展**：

- 单客户单项目：8–9 人够用
- 衍生 2–3 个变体或拓展新客户：扩到 10–12 人，主要补应用层 + MCAL

**关键考量**：

- 量产期最大的坑是**人员流失**
- 外包合同从"开发"转成"维护"模式，关键模块（Bootloader、BSW 配置）约定每年技术支持工时

### 招聘节奏一览

```
阶段          人数    新增 / 关键动作
──────────────────────────────────────────────
现状           5     —
A 样(6M)       7     测试×1, AUTOSAR接口×1
B 样(12M)      8     测试×1, 启动 FuSa 外包咨询
SOP前(18M)   8-9    应用层×1（按需）
量产维护      8-12   按业务扩展
```

## 四、外包 vs 自研的划分原则

**核心原则：不是按"难易"分，而是按"商品化 vs 雷达专属"分**

> **常见误区**：把"复杂脏活累活"外包出去
>  **正确思路**：商品化模块外包，自有 IP 和雷达 know-how 自己掌握

| 划分维度       | 外包             | 自研             |
| -------------- | ---------------- | ---------------- |
| 标准化程度     | 高（行业通用）   | 低（产品强绑定） |
| 工具链成本     | 高（不养不划算） | 中等             |
| Know-how 归属  | 通用知识         | 雷达核心资产     |
| 量产期调试需求 | 偶发             | 高频             |
| 客户审核答辩   | 可缺席           | 必须自己人       |

## 五、外包部分详解（5 大块）

### 5.1 AUTOSAR Classic BSW 完整配置

**包含模块**：COM、PduR、CanIf、CanTp、CanSM、CanNm、Dcm、Dem、NvM、Fee、EcuM、BswM、Os 配置、RTE 生成

**为什么外包**：

- 这套 stack 行业用了 15+ 年，配置模式高度固化
- 工具链（DaVinci Configurator / EB tresos）license 一年几十万，自己买不划算
- 供应商手里有大量现成模板，比从零做快 3–5 倍

**交付物要求**：

- 完整的 arxml 配置文件（可重新生成）
- 集成到自己工程的源码（**不接受只给二进制 / .a 库**）
- 配置说明文档 + 模块依赖图
- 通过 Vector CANoe 的基础通信测试报告

**合同要点**：

- 源码必须交付
- 约定 **3–6 个月驻场**支持集成调试
- 工具链 license 归属说明
- 后续每年技术支持工时（建议 200–400 小时/年）

**踩坑提醒**：

- 不要让外包碰应用层代码，边界划清楚（他们只做 BSW 到 RTE 接口为止）
- arxml 文件版本管理要进自己的 Git，不能放他们那边

------

### 5.2 Bootloader（UDS Flash Bootloader）

**包含**：UDS 诊断协议栈、Flash 驱动、应用程序跳转逻辑、安全访问（Security Access）、签名校验、A/B 分区（如果支持）

**为什么外包**：

- UDS Bootloader 高度标准化，遵循 ISO 14229 + ISO 15765 死规范
- 主机厂规范虽各有差异（吉利 GEEA、长城 GEA、上汽零束等），但模板化程度高
- 自研一个量产级 Bootloader 要 6–9 个月，外包 2–3 个月

**交付物要求**：

- Bootloader 源码 + 链接脚本
- 适配主机厂规范的诊断配置（Flash Driver、刷写流程）
- 刷写工具或刷写脚本（CANoe CAPL 或 Python）
- 完整的 UDS 测试用例和测试报告

**合同要点**：

- 必须支持目标主机厂的刷写规范（合同里写明客户名）
- 提供 EOL 产线刷写工具或对接自己的产线工具
- OTA 升级场景单独约定（如果客户要求）

**踩坑提醒**：

- 主机厂规范每年都在更新，合同要约定**至少 2 年的规范变更跟进**
- A/B 分区、回滚、版本号管理细节要 review 清楚，不同主机厂玩法不一样

------

### 5.3 MCAL 基础配置

**包含**：基础外设的 MCAL 配置（GPIO、PWM、ADC、基础 SPI、基础 CAN、Watchdog、MCU 时钟树）

**为什么外包**：

- 芯片厂（NXP S32、TI AWR、Infineon AURIX）都提供官方 MCAL，配置工作量大但技术含量低
- 供应商有现成配置参考，改改时钟树和引脚就行

**重要边界：MCAL 只外包"基础部分"**

雷达专用 MCAL 配置必须自己做（见 6.1）：

- 高速 SPI（连 RF 前端芯片）
- DMA 链表（雷达 ADC 数据搬运）
- 多芯片同步
- 高精度定时器（chirp 控制）

**交付物**：MCAL 配置 arxml + 集成验证报告

------

### 5.4 信息安全模块（SecOC / HSM）

**包含**：SecOC 协议栈、HSM 驱动、密钥管理框架、对接 EVITA 标准

**为什么外包**：

- SecOC 实现标准化，但密钥管理和 HSM 编程很专门
- 行业里就那几家做得好（ETAS、Vector 自家、个别国内供应商）
- 5–10 人团队不值当养专业信息安全人员

**交付物**：SecOC 协议栈源码 + HSM 固件 + 密钥注入流程文档

**关键边界**：

- **密钥管理流程必须自己掌握**——产线密钥注入、密钥轮换、量产密钥库管理这些不能交给外包

------

### 5.5 功能安全（部分外包）

**这是和"完全外包"最大的不同**——功能安全**不能完全外包**，必须采用"自己 owner + 外包咨询"模式。

#### 必须自己做（产品 + 责任绑定）

| 活动                             | 原因                        |
| -------------------------------- | --------------------------- |
| HARA（危害分析与风险评估）       | 必须基于产品定义            |
| FSC → TSC（功能 / 技术安全概念） | 产品强绑定                  |
| 安全需求定义和分解               | 输入错则全错                |
| 主机厂审核答辩                   | 外包不能替你出场            |
| 量产后安全问题响应               | SLA 短，外包不可能驻场 5 年 |
| DIA 协议签署                     | 法律责任主体                |

#### 可以外包（流程性、文档性、工具性）

| 活动                                                         | 说明             |
| ------------------------------------------------------------ | ---------------- |
| 详细安全分析（FMEA / FTA / DFA / FMEDA）                     | 表格活，工作量大 |
| 安全文档撰写和维护（Safety Case / Safety Manual / Safety Plan） | 模板化           |
| 工具链认证（编译器、静态分析工具的 TCL 论证）                | 一次性           |
| SEooC 论证报告                                               | 标准化           |
| 故障注入测试用例编写和执行                                   | 流程性           |
| 流程合规咨询（V 模型裁剪、追溯矩阵建立）                     | 咨询服务         |
| 认证机构（TÜV / SGS）对接和预审                              | 专业服务         |

#### 为什么不能完全外包

1. **主机厂审核必须自己答辩**：FuSa 专家会现场问 HARA 怎么做的、ASIL 等级为什么这么定，外包不在场也不熟产品
2. **DIA 责任主体唯一**：ISO 26262 要求功能安全有明确责任主体，作为雷达件开发方，最终责任由你们承担
3. **安全需求强耦合产品**：雷达漏检 / 误检的 ASIL 等级、角度精度安全失效阈值，这些只有产品团队最懂
4. **量产期安全 field issue**：主机厂要求 24–72 小时根因分析 + 8D 报告，外包工程师不可能驻场支持 5 年

#### 可以外包的供应商方向

- 经纬恒润功能安全团队
- 瑞萨 / 英飞凌 PS 服务
- 独立咨询公司（赛迪、艾普斯达等）
- TÜV / SGS 认证机构的咨询服务

------

### 5.6 工具链类（一次性外包）

- EOL 测试治具软件
- 产线刷写 / 标定工具
- Tier1 内部用的可视化调试工具

**特点**：做完就交付，不用长期维护，外包性价比高

## 六、自研部分详解（6 大块）

### 6.1 雷达专用驱动 ⭐ 核心中的核心

**包含**：

- RF 前端芯片驱动（TI AWR / NXP TEF / Calterah 等的寄存器配置）
- 高速 SPI 驱动（与 RF 芯片通信，时序敏感）
- ADC 数据采集 DMA 链路（从 RF 到 DSP/MCU 的数据搬运）
- 级联同步（多芯片方案，如 4 片 AWR2243 级联做 8×8）
- Chirp 时序控制（决定雷达性能的关键）

**为什么必须自己做**：

- 这部分 know-how 是**雷达团队的核心资产**
- 量产后调试问题（某批硬件 RF 性能漂移、级联失步）必须自己人能看寄存器查到
- 不同芯片代次切换时（如 AWR2243 → AWR2944），这部分代码迁移成本最高
- 外包不懂雷达，做出来一定不对

------

### 6.2 应用层 / 系统软件

**包含**：

- 雷达整体状态机（init → calib → normal → fault → degrade）
- 对外接口管理（CAN-FD 报文、SOME/IP 服务、Ethernet 数据流）
- 模式管理（不同工况切不同算法参数）
- 故障管理（DEM 上层应用、故障注入、降级策略）
- 时间同步（gPTP，如果用以太网）

**为什么自己做**：

- 应用层逻辑跟产品定义强相关，每个客户都不一样
- 外包不可能理解雷达的工作模式

------

### 6.3 AUTOSAR 集成（接口人角色）⭐ 关键岗位

**虽然 BSW 配置外包，但集成必须自己人 own。**

**具体职责**：

- review 外包交付的 arxml 和源码
- 集成到自己的工程，处理冲突
- 当 BSW 和应用层有问题时，定位是哪边的 bug
- 管理 AUTOSAR 版本升级（4.2.2 → 4.4 → R20-11 这种）
- 后期 BSW 维护，小改动自己改不再外包

**能力要求**：

- 必须懂 BSW 内部机制
- 能跟外包技术对话
- 通常要 5 年以上 AUTOSAR 量产经验

------

### 6.4 功能安全 owner（0.5 人，可兼职）

**为什么不能省这 0.5 人**：

这个人是功能安全在公司的"接口人 + 答辩人"，没有他外包做得再好也兜不住主机厂审核。

**职责**：

- own HARA 和 FSC（输入由产品 / 系统团队提供）
- 签 DIA、对接外包功能安全咨询公司
- 主机厂审核时出面答辩
- 量产后做安全问题归因的 owner

**人选建议**：

- 可由平台软件负责人或架构师兼任
- 项目立项时就要明确，不要拖到 B 样
- 完成 TÜV 的 FuSa Engineer 培训
- 初期可让外包咨询公司派人**驻场 3–6 个月**带入门

------

### 6.5 测试 + HIL + CI

**包含**：

- HIL 环境搭建（CANoe + VT System / dSPACE / NI PXI）
- 自动化测试用例（vTESTstudio / Python + pytest）
- CI pipeline（Jenkins / GitLab CI，每次 commit 自动跑回归）
- 实车测试支持
- 客户问题单复现和归因

**为什么自己做**：

- 测试是**量产质量底线**，外包测试覆盖不了 corner case
- 雷达特殊场景测试（特定目标距离 / 角度 / 速度组合）需要懂雷达的人设计用例
- CI pipeline 是团队效率工具，必须长期演进

**可购买（不算外包）**：

- BSW 协议测试规范包（如 Vector 的 TPK），不用自己写所有用例

------

### 6.6 RF 标定流程的软件支持（雷达专属）

**包含**：

- 暗室标定的工具软件（控制雷达发射、采集数据、写入 EEPROM）
- 产线快速标定流程
- 温度补偿表的生成和写入
- 单板自检（BIST）

**为什么自己做**：和产品强绑定，外包不懂

| **RF 标定软件**                | 自己         | 雷达专属                     |
| **密钥管理流程**               | 自己         | 安全关键                     |

## 八、外包供应商选择建议

国内适合做车载毫米波雷达 BSW 的供应商分三类：

| 类别              | 代表                                               | 适用场景                  |
| ----------------- | -------------------------------------------------- | ------------------------- |
| 大型 Tier1 服务商 | 经纬恒润、东软睿驰、普华基础软件                   | 主 BSW 打包，能力全面但贵 |
| 专项服务商        | 中科创达（系统集成）、怡和嘉业（Bootloader）       | Bootloader / 特定模块     |
| 工具厂商 PS       | Vector、ETAS、EB 的 professional service           | 技术过硬但更贵，关键模块  |
| 功能安全咨询      | 经纬恒润 FuSa 团队、瑞萨 / 英飞凌 PS、独立咨询公司 | 专项功能安全外包          |

**建议组合**：主 BSW 找一家大供应商打包 + Bootloader 找专项的（质量更好）+ 功能安全找专门咨询公司

## 十、关键落地提醒

### 10.1 外包不是"扔出去就完了"

必须有自己人 own AUTOSAR 集成（5 年以上量产经验），能：

- 看懂供应商交付的代码
- 跟供应商技术 review
- 处理集成时的冲突
- 后期成为 customer issue 的对接窗口

### 10.2 警惕"外包依赖症"

> 反面案例：某 Tier1 把 BSW 完全外包 5 年，供应商涨价后想换，发现自己人完全看不懂那套 stack，被迫整体重做，损失上千万。

**对策**：

- 每年安排自家工程师跟着外包做一轮 **code walk-through**
- 关键模块（Bootloader、BSW 核心配置）保留可接管能力
- 配置文件版本管理放自己 Git

### 10.3 招聘宁缺毋滥

**AUTOSAR 集成接口人** 和 **功能安全 owner** 这两个岗位特别难招：

- 慢慢找好的，比快速凑数强 10 倍
- 招错人量产期会反复填坑
- 优先找有相同规格量产经验的（前雷达 / ASIL B(D) 项目）

### 10.4 工具链预算要提前规划

| 工具                       | 年成本量级                        |
| -------------------------- | --------------------------------- |
| Vector CANoe + VT System   | 数十万                            |
| ETAS INCA                  | 十几万                            |
| Lauterbach Trace32         | 数十万                            |
| Vector DaVinci / EB tresos | 数十万–百万级（外包后可省一部分） |
| Polarion / Jama 需求管理   | 十几万                            |

外包 BSW 后，DaVinci / tresos 的 license 费用可以省一部分，但调试和集成工具不能省。

### 10.5 关于"自己人不能没事干"的顾虑

不用担心。外包 BSW 配置和 Bootloader 后，自己团队的活依然排得满：

- AUTOSAR 集成 review 和冲突处理
- 雷达专用驱动深度优化（外包做不了）
- 应用层状态机和对外接口（每客户不同）
- 功能安全机制实现和验证 + 主机厂答辩
- 测试自动化、HIL 用例开发、CI pipeline 维护
- 客户问题单归因、版本回归
- 多变体维护
- RF 标定流程支持

外包释放的人力应该投到 **测试自动化** 和 **雷达专用驱动深度优化**，这两块直接决定产品竞争力和量产稳定性。

### 10.6 行业参照

8×8 前雷达进入主机厂前向感知供应链，软件团队规模通常要 20+ 人才算成熟（参考博世、大陆国内团队的雷达组配置都是几十人，含算法）。

本方案的 8–9 人 BSW 团队（不含算法）属于"能跑量产但要拼命"的水平：

- 心理预期要充分
- 项目数量增长（2–3 个客户并行）后，团队还要再扩
- 不要被"看起来不忙"误导，量产期突发事件极多

## 十一、决策总结

### 应该外包的（5 大类）

1. AUTOSAR Classic BSW 完整配置
2. Bootloader / UDS Flash Bootloader
3. 基础 MCAL 配置
4. SecOC / HSM 实现（不含密钥管理）
5. 功能安全文档 / 详细分析 / 认证对接（不含 owner 责任）

### 必须自己做的（6 大类）

1. 雷达专用驱动（RF 控制、高速 SPI、DMA、级联同步、Chirp 时序）
2. 应用层 / 系统软件
3. AUTOSAR 集成（接口人角色）
4. 功能安全 owner（HARA / FSC / 主机厂答辩）
5. 测试 + HIL + CI
6. RF 标定软件支持

### 团队规模演进

- 当前：5 人
- A 样：7 人（+测试 +AUTOSAR 接口人）
- B 样：8 人（+测试，启动 FuSa 外包咨询）
- SOP：8–9 人（按需 +应用层）
- 量产维护：8–12 人（按业务扩展）

### 核心原则

> **不是"简单的自己做、复杂的外包"，而是"商品化外包、雷达 know-how 自己做"。**
>  **功能安全可以外包"脏活累活"，但产品安全责任和主机厂答辩必须自己人 own。**
