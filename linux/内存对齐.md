## **为什么需要内存对齐？**

**struct 的 alignment 规则**

	在介绍`struct`的 alignment 之前，首先需要解释什么是自然对齐（`natural alignment`）。
	
	每种数据类型都有由硬件决定的自然对齐需求，自然对齐由平台ABI + 编译器实现。需要注意的是`alignment≠size`。

struct 的 alignment 规则如下:
-	struct 本身按照成员对齐值中的最大值进行对齐
-	每个成员按照各自的自然对齐值进行对齐；为满足下一个成员的对齐，编译器会在成员之间插入 padding
例子如下：
```
// sizeof(GeneralStruct1) = 8，alignof(GeneralStruct1) = max(alignof(c),alignof(s),alignof(i)) = 4
// 以偏移量从0开始，进行 GeneralStruct1 的布局推导
// c 在 0（占 1）
// short 的对齐值为 2 ，因此需要在 s 前 padding 1 字节，使 s 起始偏移为 2，s 在 2..3（占 2）
// 现在偏移 = 4，i 需要 4 对齐，已经满足，所以 s 与 i 之间无需 padding，i 在 4..7（占 4）
// 现在偏移 = 8，且 8 是结构体对齐 4 的倍数，无需 padding
struct GeneralStruct1{
  char c;// one byte
  short s;// two bytes
  int i;// four bytes
};

// sizeof(GeneralStruct2) = 12,，alignof(GeneralStruct2) = max(alignof(c),alignof(i),alignof(s)) = 4
// 以偏移量从0开始，进行 GeneralStruct2 的布局推导
// c 在 0（占 1）
// 在 i 前 padding 3 字节，使 i 起始偏移为 4（满足 4 对齐），i 在 4..7（占 4）
// 现在偏移 = 8，s 需要 2 对齐，已满足，所以 i 与 s 之间无 padding，s 在 8..9（占 2）
// 为使 GeneralStruct2 满足 4 对齐，在末尾补 2 字节
struct GeneralStruct2{
  char c;// one byte
  int i;// four bytes
  short s;// two bytes
};

// 可以使用宏 offsetof 来查看成员的偏移，以此来验证推导是否正确，用法如：offsetof(GeneralStruct1, s)
```
**内存对齐的原因**
内存对齐的原因涉及到硬件、编译器，后文将一一介绍
-	### **硬件层面**
	硬件层面的内存对齐原因涉及到内存、缓存和处理器层面
	-	**内存层面**

​		![](.\image\memory1.png)

​	内存是按顺序排布、按字节寻址的。如果 memory 被排列成单字节宽度的单 `bank`（如上图 1 左侧所示)，那么处理器在寻址一个 	`int` 的时候需要 4 次内存周期，才能拿到这个 `int` 值。为了在一次内存周期内拿到 `int` 值，需要将 4 个单字节宽度的 `bank` 水	平排列成如上图 1 右侧所示即可。

​	在 memory 排列如上图 1 右侧所示时，假设`bank0..3` 分别有地址 `x`、`x+1`、`x+2`和`x+3`（`x`是4的倍数），此时假设一个`int` 分	配在地址`x` 上，处理器将在一个内存周期内读取这个`int`；假设为`int` 分配的地址是`y`（不是 4 的倍数），那么它将跨越两个 bank 	行，如下图 2 所示，将需要两个内存读取周期来读取。

![](.\image\memory2.png)
-	**cache层面**
  数据不对齐，可能会造成其跨越多个cache line，在使用这个数据的时候，就会使用两个cache line（一般长度是 64B），造成性能	下降。数据不对齐时，在多核并行访问下可能会造成cache 同步的消耗增加，此时需要同步两个cache line。

-	**处理器层面**

  ```
  // 在内存中紧密存储 sizeof(PackedStruct) = 7
  struct
  __attribute__((packed))
  PackedStruct
  {

        char c;
        // one byte
    
        int i;
        // four bytes
    
        short s;
        // two bytes
    };
    ```
    当结构体不内存对齐时，如上面的结构体 PackedStruct ，其在内存中的排列如下图 3 所示。

![](.\image\memory3.png)

某些处理器对内存访问有严格对齐要求，如果数据的内存不是对齐的，会触发硬件异常。即使处理器支持不对齐的数据访问，对于像上述结构体 `PackedStruct` 中的成员 `int i` 的读取，它也将先访问地址 `0x0000`（以图3 这样的内存布局为例），将其读取到 `register a`，然后往左偏移一个字节，然后访问地址 `0x0001`，将其读取到 `register b`，然后将其往左偏移三个字节，最后将前述两者进行 `OR` 操作，才能得到这个 `int` 值。
-	### **编译器层面**
	编译器在布局结构体时，会依据目标平台的ABI（应用二进制接口）规定和处理器架构的对齐约定，为每个成员安排合适的地址偏移，并插入填充字节（padding）以满足对齐要求。编译     器可以通过 __attribute__((packed)) （这个属性一般用于对象需要网络传输，以节省     带宽；或者用于某些数据库场景，以节省内存占用）等手段来改变编译器对齐规则，但是这会     影响性能，前面硬件层面已经进行了分析。
